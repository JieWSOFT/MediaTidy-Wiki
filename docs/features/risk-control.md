# 115 风控对抗

MediaTidy 内置 115 网盘风控自动化处理机制，当检测到 API 被风控限制时自动识别验证码并恢复服务。

## 工作原理

```
正常 API 调用 → 检测到 911 风控 → 自动获取验证码图片
                                        ↓
                                  ONNX OCR 识别验证码
                                        ↓
                                  提交验证结果到 115
                                        ↓
                                  风控解除，恢复调用
```

系统使用后台监控线程定期采样 115 API 调用，当检测到返回 code 911（风控触发）时，自动进入验证码破解流程。

## 启用风控防护

风控防护**默认关闭**，需要手动开启：

1. 进入「网盘配置」页面
2. 找到「115 风控防护」卡片
3. 开启总开关

::: info 前提条件
风控防护需要 OCR 模型文件（`common_old.onnx` + `charsets_old.json`，位于 `data/ocr/` 目录），Docker 镜像已内置。如果页面上看不到「115 风控防护」卡片，说明模型文件缺失或不可用。

OCR 使用 ONNX Runtime 推理，Docker 镜像的 CGo 版本已包含 ONNX Runtime 支持。非 CGo 编译版本该功能不可用。
:::

## 配置项

| 配置项 | 字段 | 默认值 | 说明 |
|--------|------|--------|------|
| **启用** | `enabled` | false | 开关风控防护功能 |
| **检查间隔** | `checkInterval` | 60 | 风控状态检查频率（秒） |
| **采样次数** | `sampleCount` | 3 | 每次检查的 API 采样调用次数（多次采样以确认是否真的被风控） |
| **最大重试** | `maxRetry` | 5 | 单次风控事件中验证码破解的最大重试次数 |

## 状态说明

每个 115 账号独立跟踪风控状态，状态机如下：

```
Normal → Detected → Solving → Resolved → Normal
                           ↘ Failed → Normal（手动介入后）
```

| 状态 | 说明 |
|------|------|
| **Normal（正常）** | API 调用正常 |
| **Detected（已检测）** | 检测到 code 911 风控 |
| **Solving（破解中）** | 正在自动识别和提交验证码 |
| **Resolved（已解除）** | 验证码破解成功，风控已解除 |
| **Failed（失败）** | 自动破解失败（达到最大重试次数），需人工介入 |

### 手动触发

在账号列表中每个账号旁有「手动检测」按钮，可主动触发一次风控检测和破解尝试。

## WAF 熔断

当检测到 HTTP 405（WAF 拦截）时，系统会自动暂停所有 115 API 调用 60 秒，避免被进一步封锁。这是比 911 风控更严重的限制，需要等待一段时间后自动恢复。

## 通知联动

风控相关事件会通过已配置的通知渠道自动推送：

- **检测到风控** → 「正在自动破解验证码...」
- **破解成功** → 「风控已解除」
- **破解失败** → 「请手动处理」

## API 接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/drive/risk-status` | GET | 获取风控状态（是否可用、是否启用、各账号状态） |
| `/api/drive/risk-config` | POST | 保存风控配置 |
| `/api/drive/risk-check/:id` | POST | 手动触发指定账号的风控检测 |
